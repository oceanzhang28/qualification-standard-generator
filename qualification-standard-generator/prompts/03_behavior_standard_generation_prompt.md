# 03 行为标准生成 Prompt

## 目标

基于用户已确认的行为模块和行为要项，生成专员、主管、经理、总监四级行为标准。

## 必读文件

- `rules/behavior_standard_writing_rules.md`
- `rules/level_difference_rules.md`
- `references/writing_examples/action_type_examples.md`
- `references/writing_examples/work_type_level_patterns.md`
- 必要时读取相近案例的 `snippets/by_behavior_item/` 和 `snippets/by_level_progression/`

## 生成要求

- 固定输出专员、主管、经理、总监四列。
- 用户未选择的职级列留空。
- 明显不承担某行为要项的职级可留空，并在校验报告或人工复核重点说明。
- 除非该职级不适用并留空，每个适用职级至少 2 条行为标准。
- 复杂行为要项可写 3-5 条，每个职级单元格最多 5 条。
- 当行为要项本身包含多个工作细项，或由多个阶段一细分要项合并而来时，必须在同一职级单元格内用 `1、` `2、` `3、` 编号展开，不得压缩成一句。
- 不强制量化，但必须可评价。
- 使用业务说明风格，具体、清楚、可复核。
- 不照搬案例内容。
- 行为动词必须体现职级角色，如参与、独立承担、主导、组织、指导、领导等。
- 不得让经理级或任何职级大量重复同一个动作词开头。
- 先判断行为要项属于哪类通用工作类型，再按 `work_type_level_patterns.md` 选择模式 A 或模式 B 的职级差异。
- 每条行为标准生成前，先判断是否存在输入、依据、场景、背景研究、业务目标、用户需求、数据、规则或协作要求等前置条件。
- 存在前置条件时，优先使用“基于/根据/通过/结合/围绕/针对/承接”等行为情景开头，再写动作、产出和评价要求。
- 不存在明确前置条件，或前置条件已由同一单元格前一条承接时，可以直接动作开头。
- 禁止整张表大量直接以动词开头。

## 生成判断

行为要项相对单一：
- 每个适用职级至少 2 条，分别对应不同动作、产出或评价重点。

行为要项包含 3 个不同动作步骤或 3 个不同工作活动：
- 每个适用职级尽量 3 条。

行为要项包含完整流程、多个工作对象、多个交付物或多个协作场景：
- 每个适用职级可 4-5 条。

## 禁止

- 不得写学历、年限、经验、通用素质、领导力、价值观。
- 不得写“具备良好的沟通能力”等空泛能力。
- 不得为凑数量制造重复句。
- 不得将不成熟案例中的低颗粒度行为标准当成优质样例。
