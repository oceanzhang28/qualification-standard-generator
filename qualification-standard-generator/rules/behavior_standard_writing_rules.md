# 行为标准写作规则

## 基本原则

行为标准必须是业务人员能理解、HR 能评估、部门能复核的专业工作项描述。

语言风格采用业务说明风格：
- 清楚。
- 具体。
- 容易让业务人员理解。
- 保持专业和可评价。
- 避免过度制度化和空泛官话。

## 四个组成部分

每条行为标准应隐含包含四个部分，但最终输出为自然完整的行为描述，不拆成四栏。其中最核心的是“行为方式”，因为它体现该职级在该工作中的职责角色。

| 组成部分 | 说明 |
|---|---|
| 行为情景 | 行为发生时的场景，如“在……情况下”“根据……”；可选 |
| 行为内容 | 做什么，是行为过程和行为路径；必选 |
| 行为方式 | 怎么做，使用体现职责角色的动词；必选 |
| 行为结果 | 完成后形成什么产出、效果、状态或交付结果 |
| 衡量标准 | 输出成果的质量和形式要求，可量化也可定性 |

推荐句式：

```text
基于【输入/依据/场景】，完成【行为动作】，形成【产出/结果】，确保【衡量标准】。
```

可以变化句式，但不得丢失行为方式、行为内容、产出和评价要求。

## 行为情景使用规则

多数行为标准应先判断是否存在前置条件。前置条件可以是输入、依据、场景、背景研究、业务目标、用户需求、数据、规则、流程节点或协作要求。

当行为要项存在明确前置条件时，行为标准应优先使用行为情景开头，再写行为动作、结果和评价要求。

常用行为情景开头：

```text
基于……
根据……
通过……
结合……
围绕……
在……情况下
针对……
承接……
```

示例：

```text
基于产品企划输入和市场洞察资料，独立完成系列设计主题、风格定位和款式结构规划，形成设计企划方案，确保设计方向能够支撑后续样品开发。
```

不是所有行为标准都必须有行为情景。以下情况可以直接以动作开头：
- 行为本身是简单执行或固定 SOP。
- 前置条件在同一单元格前一条中已经明确，后续编号承接同一情景。
- 强行添加情景会导致句子冗长或虚假。

禁止所有行为标准都直接以动词开头。生成时应优先判断是否需要行为情景，并避免同一单元格中每条都使用完全相同的开头。

## 行为动词角色

行为动词必须反映该职级承担的角色和贡献，不能只作为普通动词使用。

| 动词 | 角色含义 |
|---|---|
| 在他人指导下 | 还不能独立完成，需要关键指导 |
| 参与 | 作为团队成员独立完成部分工作，但不是核心或主要成员 |
| 配合 / 协助 | 为主责人提供资料、执行、分析、跟进等支持 |
| 独立承担 | 工作被分配后，能按要求独立完成完整任务或明确子任务 |
| 主导 | 工作核心内容由本人完成，对核心产出负责 |
| 组织 | 类似项目负责人，承担组织协调和一定指导责任，对工作结果负责 |
| 组织并作为核心成员 | 不仅组织协调，同时承担核心专业工作 |
| 指导 | 工作由他人完成，本人确定方向、原则和思路，并帮助解决关键难题 |
| 领导 | 等同“组织 + 指导”，同时包含施加影响和推动决策的含义 |

使用提醒：
- 不要把经理级默认写成“统筹”。
- 总监和经理也可能出现“配合”“参与”等动作，例如配合公司级或跨部门工作；动作选择必须根据该职级在具体工作中的真实角色判断。
- 同一职级在同一张表中不得高频重复同一个动作词开头。

## 评价标准

行为标准必须可评价，但不强制量化。

可使用量化评价：

```text
及时率、准确率、完成率、覆盖率、合规率、达成率、采用率、差错率
```

也可使用定性评价：

```text
判断清晰
逻辑完整
方案可落地
能支撑决策
风险识别充分
过程闭环
输出内容可被业务采纳
```

禁止无法评价的空话：

```text
持续提升能力
增强意识
积极配合
提高水平
具备能力
熟悉流程
```

## 数量和颗粒度规则

- 除非该职级不适用并留空，否则每个适用职级至少 2 条行为标准。
- 行为要项可以拆成 3 个不同动作步骤，或可以拆成 3 个不同工作活动项时，尽量写 3 条。
- 当行为要项包含完整流程、多个工作对象、多个交付物或多个协作场景时，可拆至 4-5 条。
- 每个职级单元格内最多 5 条，不超过 5 条。
- 如果行为要项本身包含多个工作细项，或由多个阶段一细分要项合并而来，即使模块精简，也必须在行为标准单元格内用 `1、` `2、` `3、` 编号展开。
- 可拆分流程，或行为要项是一个类型的工作合并时，必须把各种类型工作拆分开写。
- 不制造重复或换词式标准。
- 不为了凑数量制造无意义拆分；每条编号必须对应不同动作步骤、工作活动、交付物或评价重点。

## 行为要项合并后的展开规则

精简框架中的行为要项常常承接多个细分工作。生成行为标准时，不能因为行为要项变宽就删除细项。

示例：

```text
行为要项：产品设计
可展开为：
1、设计方向和款式方案
2、设计图稿和细节表达
3、工艺版师协同和样品跟进
4、样品调整和设计方案优化
```

行为标准应在同一职级单元格内按这些细项编号，而不是压缩成一句。

## 句式变体

推荐句式不是唯一句式。应根据动作类型选择不同表达。

- 分析类：基于市场、竞品、数据或用户反馈，开展分析、诊断或洞察，输出分析报告或策略建议。
- 计划类：结合业务目标和历史执行情况，拆分任务、规划资源和节点，支持计划推进。
- 设计/研发类：根据产品设计或研发需求，开展设计、打样、测试、评估或评审，输出设计方案、样品、数据文件或评审结论。
- 推进类：根据业务阶段和任务要求，推动方案落地、过程协同和问题闭环，支撑目标达成。
- 协同类：围绕跨部门需求，完成资料输入、需求澄清、协作推进和结果反馈，确保协作事项闭环。
- 管理类：围绕干部、人才、项目、成本、质量或时序等对象，开展评估、计划、跟进、调整和决策支持。
- 机制建设类：针对流程、机制或效率问题，制定或优化规则、模板、流程和方法，推动复制应用。

## 禁止写法

以下写法不符合本公司任职资格标准：

```text
具备良好的沟通能力
有较强的数据分析能力
能够积极主动完成工作
熟悉相关业务流程
具备团队管理意识
提升工作效率
完成领导交办事项
```

## 空白职级处理

默认尽量为用户选择的适用职级生成行为标准。

只有当该行为要项明显不属于某职级职责范围时，才允许该职级单元格留空，并在校验报告或人工复核重点说明原因。
