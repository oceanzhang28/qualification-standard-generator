# 行为模块与行为要项拆解规则

## 拆解目标

根据岗位关键任务，拆解出行为模块和行为要项，形成任职资格标准的结构框架。

阶段一只生成行为模块和行为要项，不生成行为标准。

## 拆解方式

### 流程化拆解

适合流程清晰、工作链条明确的岗位。

常见模块：

```text
需求输入与分析
方案规划与制定
资源协调与推进
执行监控与问题处理
结果复盘与优化
标准沉淀与机制建设
```

### 活动化拆解

适合工作活动并行、专业领域多元的岗位。

常见模块：

```text
数据分析
计划管理
项目推进
跨部门协同
风险控制
机制建设
```

### 混合拆解

当岗位既有清晰流程，又有多个专业活动时，可采用流程化 + 活动化结合。

生成时必须说明：
- 采用哪种拆解逻辑。
- 行为模块边界。
- 如何保证 MECE。
- 用户关键任务覆盖关系。

## 数量规则

- 组织贡献以外的岗位专业模块默认 4-6 个。
- 每个岗位必须包含固定模块：组织贡献。
- 组织贡献固定包含 2 个行为要项：平台/流程建设、人员培养。
- 除组织贡献外，每个行为模块默认 3-5 个行为要项。
- 如果岗位明显复杂到需要超过 6 个专业模块，不得自行扩展，应提示用户拆分岗位、收窄范围或确认扩展。
- 如果用户要求精简，可以将相邻流程模块合并为更少的行为模块，但必须保留关键任务覆盖关系。
- 精简时优先合并：
  - 策略、洞察、企划、设计等同属产品开发主流程的模块。
  - 成本、时序、质量等同属过程管理的模块。
  - 创新研究、专项推进等项目型工作可合并为项目管理模块。
- 精简时不得删除固定组织贡献模块。

## 命名规则

行为模块命名：
- 使用工作领域名称。
- 避免能力词、态度词、结果指标词。
- 示例：销售管理、数据分析、商品运营、产品研发、业务赋能。

行为要项命名：
- 使用具体工作项、关键步骤或关键产出。
- 示例：目标规划、市场洞察、产品开发、风险管控、平台/流程建设。

## 固定组织贡献模块

所有岗位都必须包含：

```text
行为模块：组织贡献
行为要项：平台/流程建设
行为要项：人员培养
```

如果公司当前岗位案例将平台建设和流程建设拆成两个行为要项，可在该岗位中拆为“平台建设”“流程建设”“人员培养”；但生成新岗位时默认优先使用统一规则“平台/流程建设、人员培养”，除非用户明确要求拆分。

该模块属于统一要求，不因用户关键任务未显式提及而删除。

## 关键任务覆盖

阶段一输出必须包含关键任务覆盖关系：

```text
关键任务 1 -> 行为模块 X / 行为要项 Y
关键任务 2 -> 行为模块 X / 行为要项 Z
```

如果某项关键任务无法归入任何模块，必须提示用户补充或调整岗位边界。
