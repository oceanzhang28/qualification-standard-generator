# 岗位访谈提炼 Skill 与任职资格生成 Skill 对接设计

## 1. 设计目标

在现有项目中新增一个独立的岗位访谈提炼 Skill，与现有 `qualification-standard-generator` Skill 组成两阶段工作流：

1. 岗位访谈提炼 Skill 从 HR 基本信息表和访谈转写稿中提炼岗位事实；
2. 人工审核岗位事实和关键任务；
3. 任职资格生成 Skill 基于审核后的岗位信息生成任职资格标准。

本设计只确定两个 Skill 的职责边界、信息交接格式和人工确认机制，不修改现有 Skill，也不包含新 Skill 的具体实现计划。

访谈方法、90 分钟提纲和转写提炼规则见 `2026-07-17-position-key-task-interview-design.md`。

## 2. 现有任职资格生成 Skill 回顾

### 2.1 能力范围

现有 Skill 面向单个专业类岗位，具备以下能力：

- 校验职类、职务子类、所在部门、岗位名称和适用职级；
- 检查 5–10 条岗位关键任务的颗粒度；
- 收集主要工作对象、主要工作流程和主要产出物；
- 基于岗位信息生成行为模块和行为要项；
- 检查关键任务覆盖关系和 MECE；
- 固定生成“组织贡献”模块及“平台/流程建设”“人员培养”两个行为要项；
- 经用户确认阶段一框架后，生成专员、主管、经理、总监四级行为标准；
- 按责任范围、问题复杂度、影响范围、决策权限和资源协调范围形成职级递进；
- 校验结构、内容、职级区分度、MECE 和输出格式；
- 输出 Markdown 预览、校验报告和 Excel 可复制 TSV。

现有 Skill 不处理学历、年限、通用素质、领导力、价值观，也不处理管理岗位任职资格体系。

### 2.2 现有流程

```text
用户提交结构化岗位信息
        ↓
检查必填信息和关键任务质量
        ↓
阶段一：行为模块与行为要项
        ↓
用户确认阶段一框架
        ↓
阶段二：四职级行为标准
        ↓
自动校验与修订
        ↓
Markdown + 校验报告 + TSV
```

### 2.3 当前输入假设

现有 Skill 假设用户已经能够提供结构化、基本可信的岗位信息。它可以检查信息质量和整理低颗粒度任务，但不负责：

- 从长篇访谈转写稿中还原岗位事实；
- 区分专家原意、AI 归纳和系统预测；
- 处理访谈中的重复、矛盾、口语、省略和协作边界混淆；
- 组织岗位事实的人工审核过程。

因此，访谈提炼不应作为现有 Skill 内部的一个普通输入步骤。

## 3. 总体架构

项目采用同一仓库内两个平级 Skill：

```text
任职资格AI生成工具/
├── position-interview-extractor/       # 新增：岗位访谈提炼 Skill
├── qualification-standard-generator/  # 现有：任职资格生成 Skill
├── docs/
├── README.md
└── AGENTS.md
```

两个 Skill 独立触发、独立运行，不直接读取对方内部提示词和规则文件。二者只通过约定的岗位信息包交接。

### 3.1 新 Skill 的职责

新 Skill 负责：

- 接收 HR 岗位基本信息表和完整访谈转写稿；
- 提取岗位定位、工作对象、流程、产出和责任边界；
- 识别、归并并标准化 5–10 条关键任务；
- 提取不同作用范围下的专业工作职责与要求差异；
- 标记事实来源、信息缺口和冲突；
- 输出岗位事实审核稿；
- 在人工确认后输出可提交生成 Skill 的清洁岗位信息包。

新 Skill 不负责：

- 生成行为模块和行为要项；
- 生成四职级行为标准；
- 将系统通用职级规则伪装成专家事实；
- 跳过人工审核直接调用生成 Skill。

### 3.2 现有 Skill 的职责

现有 Skill 继续负责：

- 使用审核后的岗位信息；
- 生成和调整阶段一框架；
- 在用户确认后生成阶段二行为标准；
- 使用专家确认的职级差异约束生成；
- 对缺失部分使用内置通用职级规则推演；
- 校验、修订并输出最终结果。

## 4. 人与 AI 的协作边界

| 角色 | 主要责任 | 不应承担 |
|---|---|---|
| 岗位专家 | 描述真实工作、具体案例、职责边界和熟悉职级的实际差异 | 推测不熟悉职级、直接编写任职资格语言 |
| HR 访谈者 | 追问任务闭环、复杂情形、协同边界和遗漏内容 | 代替专家创造业务事实 |
| 访谈提炼 Skill | 抽取、归并、规范表达、标记来源和缺口 | 虚构事实、生成任职资格标准 |
| 人工审核者 | 确认关键任务、边界和职级事实是否准确 | 要求所有任务必须具备完整四级差异 |
| 任职资格生成 Skill | 基于已确认事实完成专业拆解和规则推演 | 将未确认的访谈内容当作确定事实 |

## 5. 为什么采用分层稀疏的职级差异

访谈中获得的职级差异可能具有三种作用范围：

1. 岗位整体：描述整个岗位四级的总体责任、复杂度和影响范围；
2. 工作领域或任务组：只适用于一组相近任务，如规划类、分析类、执行推进类任务；
3. 单项任务：只适用于某项具体任务。

专家不一定能够针对每项任务分别描述四个职级，也不一定熟悉全部职级。因此：

- 不强制每项任务设置完整四级矩阵；
- 不强制四个职级全部有内容；
- 没有信息不等于职级不适用；
- 不用通用规则在事实审核阶段填满空白；
- 生成阶段才对缺失部分使用内置规则推演。

字段名称统一使用“专业工作职责与要求差异”，不使用容易引向通用素质的“能力差异”。

## 7. 最低有效输入

清洁岗位信息包的必填项为：

- 审核状态为“已确认”；
- 职类、职务子类、所在部门、岗位名称和适用职级范围；
- 5–10 条已确认关键任务；
- 主要工作对象；
- 主要工作流程；
- 主要产出物。

以下信息尽量提供，但不阻断交接：

- 岗位核心价值；
- 主要协作方和职责边界；
- 典型复杂场景；
- 关键业务指标。

以下信息完全选填：

- 岗位整体职级差异；
- 工作领域或任务组职级差异；
- 单项关键任务职级差异。

若未获得可靠的职级差异，必须明确写明：

```text
【已确认的专业工作职责与要求差异】
本次访谈未获得可靠的职级差异信息。

【职级差异信息说明】
专家主要熟悉职级：
本次获得差异的范围：未获得
未获得信息的职级、任务组或任务：全部
允许生成 Skill 使用内置通用职级规则补充：是
```

不得以空白代替缺失说明。

## 8. 生成 Skill 的信息使用优先级

生成 Skill 在相关任务范围内按以下优先级使用职级信息：

```text
人工确认的单项任务差异
        >
人工确认的工作领域或任务组差异
        >
人工确认的岗位整体差异
        >
系统内置通用职级规则
```

使用规则：

1. 上层差异只作为下层生成的总体约束，不机械复制到所有任务；
2. 任务组差异只能用于明确列出的相关任务；
3. 单项任务差异只用于该任务；
4. 某个职级有专家信息、其他职级缺失时，只对缺失职级使用通用规则；
5. 已经人工确认的岗位事实与通用规则冲突时，优先使用岗位事实，并在人工复核重点中说明；
6. 若用户不允许系统补充职级差异，生成 Skill 应提示信息不足，而不是自行预测；
7. 不得因为某一职级信息缺失而自动判断该职级不适用。

## 9. 防止错误继承

每组职级差异必须绑定作用范围：

```text
差异主题：
适用层级：岗位整体 / 工作领域或任务组 / 单项任务
适用关键任务：
明确不适用的任务：
职责与要求差异：
信息来源：
人工确认状态：
```

例如，专家说明“经理负责策略制定”，如果该描述只针对规划类任务，就不能让经理在执行、复盘或日常维护任务中全部“制定策略”。

## 10. 人工确认关口

两个 Skill 之间设置强制人工关口：

```text
岗位事实审核稿
        ↓
HR 或岗位专家修改、确认
        ↓
审核状态 = 已确认
        ↓
生成清洁岗位信息包
        ↓
允许提交任职资格生成 Skill
```

以下任一情况存在时，不应标记为“已确认”：

- 关键任务少于 5 条；
- 岗位主责与协作方责任存在重大冲突；
- 主要工作对象、流程或产出物缺失；
- 关键任务存在明显重复或遗漏且尚未处理；
- 岗位基本信息与访谈内容存在未解决冲突。

职级差异缺失不影响审核通过，只需明确缺失范围和是否允许生成 Skill 使用内置规则。

## 11. 兼容现有 Skill 的方式

短期内，清洁岗位信息包可以映射到现有 `config/user_input_form.md`：

- 岗位基本信息映射到现有必填字段；
- 5–10 条已确认任务映射到“岗位关键任务”；
- 工作对象、流程和产出物映射到“岗位特殊说明”；
- 协作方、典型场景、指标映射到现有可选字段；
- 分层职级差异整体放入现有“特殊职级差异”可选字段。

这样可以先建设和测试访谈提炼 Skill，而不立即修改现有 Skill。

后续验证稳定后，再考虑让现有 Skill 显式识别：

- 审核状态；
- 岗位整体职级差异；
- 工作领域或任务组差异；
- 单项任务差异；
- 是否允许使用内置通用职级规则补充。

## 12. 验收标准

至少选取 3 个不同工作类型的专业岗位进行端到端测试：

1. 访谈提炼 Skill 能输出 5–10 条可审核的关键任务；
2. 专家只提供整体或部分职级差异时，输出仍然完整且不会强行补齐事实；
3. 每条职级差异都有明确作用范围；
4. 信息缺失不会被错误解释为职级不适用；
5. 人工审核完成后，清洁岗位信息包能够映射到现有输入表单；
6. 现有生成 Skill 能利用已提供差异，并对其余部分使用内置规则；
7. 生成结果能够区分岗位事实约束与系统通用推演；
8. HR 审核岗位信息包原则上不超过 15 分钟；
9. 两个 Skill 的职责没有发生越界。

## 13. 后续实施边界

实施时应先新建 `position-interview-extractor` Skill，并完成独立测试，再评估是否需要修改现有生成 Skill。

第一版新 Skill 只需要支持：

- 岗位基本信息表；
- 访谈转写稿；
- 岗位事实审核稿；
- 人工确认；
- 清洁岗位信息包。

第一版不需要：

- 自动调用任职资格生成 Skill；
- 自动处理音视频转写；
- 批量并发处理多个岗位；
- 直接生成 Excel；
- 建立外部数据库或管理后台。
