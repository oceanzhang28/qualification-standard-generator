# qualification-standard-generator 开发实现流程规划

## 1. 项目目标

建设一个名为 `qualification-standard-generator` 的对话式 Skill，用于生成公司岗位任职资格标准初稿。

本项目中的“任职资格标准”不是学历、年限、通用素质、领导力或价值观要求，而是面向专业工作项的岗位工作标准。每条行为标准必须体现实际工作任务、具体行为动作、工作产出和可评价标准。

## 2. v0.1 范围

### 2.1 包含内容

- 单次生成一个岗位的任职资格标准。
- 面向 HR/OTD 专业用户，以及已由 HR 完成前置沟通、理解公司任职资格模式的业务人员。
- 固定使用四个职级：专员、主管、经理、总监。
- 公司任职资格体系仅适用于专业类岗位，不适用于管理岗位。
- 输入字段采用公司职位图谱口径：职类、职务子类；职位类别字段在 v0.1 中不适用。
- 所有岗位固定包含“组织贡献”行为模块，并固定拆分为“平台/流程建设”和“人员培养”2 个行为要项。
- 默认采用分阶段生成：
  1. 收集岗位信息。
  2. 生成行为模块和行为要项。
  3. 等待用户确认框架。
  4. 生成各职级行为标准。
  5. 自动校验并修订。
  6. 输出 Markdown 预览表、校验报告、关键修订说明、人工复核重点和 TSV。
- 输出固定 A-G 列结构：
  - A：序号
  - B：行为模块
  - C：行为要项
  - D：专员行为标准
  - E：主管行为标准
  - F：经理行为标准
  - G：总监行为标准
- 提供案例库结构和 Excel 转 Markdown 转换能力。

### 2.2 不包含内容

- 不批量生成全部岗位。
- 不直接生成 `.xlsx` 文件。
- 不建设 Web 表单。
- 不开放给完全不了解公司任职资格模式的普通用户。
- 不生成传统任职条件、学历、年限、素质能力、领导力模型或价值观要求。
- 不在 v0.1 接入 OpenClaw，仅为后续迁移预留结构。

## 3. 已确认规则

### 3.1 使用对象

v0.1 使用对象为 HR/OTD 专业用户，以及已由 HR 做过初步沟通、理解公司任职资格模式的业务人员。

因此 Skill 不需要大量解释任职资格基础概念，但必须严格执行结构、质量和校验规则。

### 3.2 职级规则

- 固定四级：专员、主管、经理、总监。
- 最终表格始终保留四个职级列。
- 如果用户选择的适用职级不是完整四级，未选择的职级列留空。
- 默认尽量为适用职级生成行为标准。
- 只有当某行为要项明显不属于某职级职责范围时，才允许该职级单元格留空，并在校验报告或人工复核重点中说明。

### 3.3 职位图谱字段规则

- 公司制定任职资格的岗位均为专业类岗位，管理岗位不适用本任职资格体系。
- 用户输入和案例元数据中使用“职类”和“职务子类”两个字段。
- 原规划中的“职族”统一理解为专业类岗位属性，不作为用户需要反复判断的字段。
- 原规划中的“职位类别”在 v0.1 中标记为“不适用”，不作为必填字段。
- 项目参考文件为：`任职资格参考标准/职位图谱.xlsx`。

### 3.4 岗位关键任务规则

- 用户必须提供 5-10 条岗位关键任务。
- Skill 在要求用户填写关键任务前，应给出合格和不合格示例。
- 每条关键任务应尽量包含：动作 + 对象/事项 + 产出或目的。
- 如果用户输入的是一段自由描述，Skill 可以先整理为 5-10 条候选关键任务，但必须让用户确认后才能继续。
- 如果关键任务过短、过泛或只有职责名词，Skill 不直接生成，应先自动改写一版并请用户确认。

### 3.5 行为模块规则

- 每个岗位默认生成 4-6 个行为模块。
- 行为模块应是岗位关键工作领域，不能写成通用能力词。
- 如果关键任务明显复杂到需要超过 6 个模块，Skill 不自动扩展，应提示用户确认是否拆分岗位、收窄范围，或明确允许扩展。
- 所有岗位必须包含固定行为模块“组织贡献”。
- “组织贡献”模块固定包含 2 个行为要项：平台/流程建设、人员培养。
- “组织贡献”属于统一要求，不因岗位关键任务未显式提及而删除。
- 生成和校验时，除“组织贡献”外的岗位专业模块仍按 4-6 个的默认范围控制；如果后续希望将固定模块计入总模块数，需要单独调整规则。

### 3.6 行为要项规则

- 每个行为模块下默认 3-5 个行为要项。
- 少于 3 个时，检查模块是否过窄。
- 多于 5 个时，检查模块是否过宽。
- 优先通过合并、拆分或调整模块边界解决数量异常。
- “组织贡献”模块是例外，固定为 2 个行为要项，不因少于 3 个而判定为异常。

### 3.7 行为标准规则

- 每个行为要项下，每个适用职级生成 1-3 条行为标准。
- 简单、执行型要项默认 1 条。
- 中等复杂要项默认 2 条。
- 涉及分析、机制、跨部门推进、风险控制的复杂要项可生成 2-3 条。
- 禁止为了凑数量制造重复或空泛标准。
- 行为标准语言采用业务说明风格：清楚、具体、易于业务人员理解，同时保持专业和可评价。
- 行为标准必须可评价，但不强制量化。可使用定性评价，如判断清晰、逻辑完整、方案可落地、能支撑决策、风险识别充分、过程闭环等。

### 3.8 分阶段确认规则

- 阶段一只生成行为模块和行为要项，并说明拆解逻辑、模块边界和关键任务覆盖关系。
- 阶段一后必须等待用户确认。
- 用户回复“确认”“继续”“没问题”等，才进入阶段二。
- 用户提出修改意见时，只修改阶段一框架，不生成行为标准。
- v0.1 最多支持 2 轮框架修改；超过 2 轮时提示用户先人工收敛岗位边界或关键任务。

### 3.9 校验与修订规则

- 生成后必须自动校验并修订。
- v0.1 不保留完整修订前版本。
- 最终只输出自动修订后的版本，同时说明主要问题、修订动作和人工复核重点。

### 3.10 最终输出顺序

1. 拆解逻辑说明
2. Markdown 预览表
3. 校验报告
4. 关键修订说明
5. 人工复核重点
6. TSV 可复制版

TSV 必须放在最后，使用单独代码块，便于直接复制到 Excel。

## 4. 开发步骤

### 第一步：冻结 Skill 设计

目标：形成 v0.1 的固定边界和成功标准。

需要完成：
- 确认 Skill 名称：`qualification-standard-generator`。
- 确认使用对象、适用范围和不适用范围。
- 确认最终输出结构和职级列规则。
- 确认分阶段生成流程。

验收标准：
- 有一份清晰的开发实现流程规划。
- 有一份材料清单。
- 后续开发 Skill 文件包时，不需要重新讨论基础边界。

### 第二步：建立制度底座文件

目标：沉淀生成质量最关键的规则文件。

优先建设 4 个规则文件：
- `rules/qualification_definition.md`
- `rules/level_difference_rules.md`
- `rules/behavior_standard_writing_rules.md`
- `rules/validation_and_revision_rules.md`

验收标准：
- 每个规则文件都能直接指导 AI 生成或校验。
- 规则包含正例、反例和处理动作。
- 不只写原则，必须可执行。

### 第三步：建立输入表单和追问规则

目标：让 Skill 能稳定收集岗位信息。

需要定义：
- 必填字段。
- 可选字段。
- 职位图谱字段校验规则。
- 信息缺失时的追问规则。
- 关键任务质量判断和自动改写确认机制。

验收标准：
- 用户信息不足时，Skill 不直接生成。
- 用户输入杂乱描述时，Skill 能先整理为任务清单并请求确认。

### 第四步：建立生成工作流 Prompt

目标：把完整任务拆成稳定的阶段性 Prompt。

建议文件：
- `prompts/01_intake_prompt.md`
- `prompts/02_framework_generation_prompt.md`
- `prompts/03_behavior_standard_generation_prompt.md`
- `prompts/04_validation_revision_prompt.md`
- `prompts/05_output_formatting_prompt.md`

验收标准：
- 每个 Prompt 只负责一个阶段。
- 阶段一不生成行为标准。
- 阶段二基于已确认框架生成行为标准。
- 校验 Prompt 能输出问题清单并自动修订。

### 第五步：建立模板文件

目标：保证输出格式稳定。

建议文件：
- `templates/stage_1_framework_template.md`
- `templates/stage_2_standard_table_template.md`
- `templates/markdown_preview_template.md`
- `templates/tsv_output_template.md`
- `templates/validation_report_template.md`

验收标准：
- Markdown 预览表字段固定。
- TSV 字段顺序固定。
- TSV 单元格内多条行为标准使用 `1、` `2、` `3、` 编号。
- TSV 前后不夹杂解释文字。

### 第六步：建立写法示例库

目标：先用高质量样例稳定行为标准写法。

第一批示例按 10 类行为动作整理：
1. 分析类
2. 计划类
3. 执行类
4. 推进类
5. 协同类
6. 监控类
7. 复盘类
8. 机制建设类
9. 风险控制类
10. 标准制定类

每类需要包含：
- 适用场景
- 推荐动作词
- 不推荐动作词
- 通用句式
- 专员、主管、经理、总监四级示例
- 常见错误写法

验收标准：
- 至少形成 40 条高质量分职级示例。
- 示例体现专业工作项，不写通用素质能力。

### 第七步：建立成熟案例库

目标：将首批成熟或可参考岗位案例转成 AI 适合读取的 Markdown 案例库。

第一批成熟或可参考案例：
- HRBP
- 有纺产品企划
- 产品研发
- 产品内容策划
- 电商运营

补充对照案例：
- 产品设计岗：版本不完全成熟，可用于学习精简模块结构、产品设计类工作细项拆分和测试迭代，不把低颗粒度行为标准当作优质写法样例。

案例库结构：
- `references/case_library/full_positions/`
- `references/case_library/snippets/by_module/`
- `references/case_library/snippets/by_behavior_item/`
- `references/case_library/snippets/by_level_progression/`

验收标准：
- 一个 Excel 文件对应一个岗位案例。
- 案例元数据使用职类、职务子类、所在部门、岗位名称、适用职级范围。
- 保留原始 A-G 列内容。
- 空白职级单元格保持为空，不自动补写。
- 每个完整案例拆分出模块、要项和职级递进片段。
- 案例质量标签必须区分“成熟”和“行为模块与要项成熟、行为标准不成熟”，避免 AI 错误学习低颗粒度行为标准。

### 第八步：开发 Excel 转 Markdown 脚本

目标：降低后续案例库维护成本。

脚本建议位置：
- `scripts/excel_to_markdown_converter.py`

脚本要求：
- 读取统一 A-G 列 Excel。
- 一个文件输出一个完整岗位 Markdown 案例。
- 同时拆出行为模块、行为要项、职级递进片段。
- 输出 YAML metadata。
- 缺失值处理为空字符串。
- 保留原始编号。
- 不改变行为标准内容。
- 文件名自动清洗非法字符。
- 如果输入列结构不符合 A-G 模板，停止并提示错误，不猜测字段含义。

验收标准：
- 能成功转换首批 5 个案例和产品设计岗对照案例。
- 转换后 Markdown 内容与 Excel 原文一致。
- 空白单元格不会被自动填充。
- 能读取并保留案例评价信息，包括最值得学习的地方、不建议学习的地方、特殊业务背景和适合迁移范围。

### 第九步：组装 Skill 文件包

目标：形成可在 Codex 中使用、后续可迁移到 OpenClaw 的 Skill 文件包。

建议结构：

```text
qualification-standard-generator/
├── SKILL.md
├── agents/
│   └── openai.yaml
├── config/
├── rules/
├── prompts/
├── templates/
├── references/
│   ├── writing_examples/
│   └── case_library/
└── scripts/
    └── excel_to_markdown_converter.py
```

说明：
- 不在 Skill 包内放 `README.md`，避免违反 Codex skill 简洁原则。
- 面向使用者的说明可放入 `SKILL.md` 或 `references/usage_notes.md`。

验收标准：
- `SKILL.md` 能作为总入口。
- 详细规则放在子文件中，按需读取。
- 文件结构清晰，便于后续迁移到 OpenClaw。

### 第十步：测试与迭代

目标：验证 Skill 是否能稳定生成可用初稿。

建议测试三类岗位：
- 结构清晰岗位：如 HRBP、薪酬管理、计划运营。
- 协同复杂岗位：如电商运营、品类运营、项目管理。
- 专业模糊岗位：如产品内容策划、组织发展、品牌策划。
- 固定模块测试：检查每个测试岗位是否都包含“组织贡献”模块及其 2 个固定要项。

每次测试记录：
- 输入信息。
- 阶段一框架结果。
- 用户修改意见。
- 阶段二生成结果。
- 校验问题。
- 人工修订点。
- 需要回写到规则库或示例库的内容。

验收标准：
- 至少完成 3 个不同类型岗位测试。
- 每次测试后能明确改进规则、示例或案例库。
- 最终输出能直接复制到 Excel 统一模板。

## 5. v0.1 成功标准

v0.1 完成时，应具备以下结果：

- `qualification-standard-generator` Skill 文件包可用。
- 用户能通过标准表单输入单个岗位信息。
- Skill 能分阶段生成行为模块、行为要项和四级行为标准。
- Skill 能按职位图谱字段识别职类和职务子类。
- Skill 能在所有岗位中固定生成“组织贡献”模块及其 2 个固定要项。
- Skill 能自动校验并修订。
- Skill 能输出 Markdown 预览表和 TSV。
- 首批 5 个成熟或可参考案例已转换为 Markdown 案例库，产品设计岗对照案例已单独标记质量边界。
- 已有 10 类行为动作写法示例。
- 至少完成 3 个岗位测试并记录迭代点。

## 6. 后续版本方向

v0.1 稳定后，可考虑：

- 扩展更多岗位序列特殊职级差异规则。
- 增加岗位序列常见模块拆解框架。
- 批量导入更多成熟案例。
- 生成 `.xlsx` 文件。
- 接入 OpenClaw。
- 增加 HR 与业务部门共创修订流程。
- 与晋升、培训、人才盘点和岗位图谱衔接。
